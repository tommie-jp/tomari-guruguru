# リポジトリ名の変更（リネーム）チェックリスト

いずれ `tomari-guruguru` から別名にリネームする予定。そのときの影響範囲・手順・
upstream マージとの関係をまとめたメモ。

関連: [02-フォーク元の取り込み.md](02-フォーク元の取り込み.md) / [01-使い方.md](01-使い方.md)

## 結論

- フォークでもリネームは自由。**フォーク関係も `upstream` からの pull も壊れない**。
  GitHub が旧URL→新URLのリダイレクトを張る。
- ただしこのプロジェクトは **GitHub Pages のベースパス `/tomari-guruguru/`** が
  コード中に埋め込まれている。リネームすると Pages URL が
  `tommie-jp.github.io/<新名>/` に変わるので、base 系の更新と再デプロイが必須。
- GitHub に「表示名」は無い。**リポジトリ名＝プロジェクト名**。

## 壊れるもの・壊れないもの

- 壊れない: フォーク関係、`upstream` remote、star / issue / PR、ローカル clone
  （いずれも GitHub のリダイレクトで追従）。
- 要更新: GitHub Pages の URL（`tommie-jp.github.io/<新名>/`）と、それに紐づく
  base path 群（下記チェックリスト）。

## 更新が必要なファイル（チェックリスト）

リネーム時に `/tomari-guruguru/` を新リポジトリ名へ置き換える。

- [ ] `vite.config.js` … `base`（最重要。ビルドの公開パス）
- [ ] `package.json` … `preview` の `--base /tomari-guruguru/`
- [ ] `vite.fork.js` … preview で開く `open` パス
- [ ] `scripts/verify-pages-build.mjs` … `BASE` 定数と検証用の正規表現
- [ ] `*.html` の OGP / Twitter メタ（`index` / `talk` / `guruguru` / `camera` /
      `tracking` の `og:url` / `og:image` など）
- [ ] `README.md` … デモリンク・ベースパスの記述
- [ ] `docs-camera/*.md` … 本番URLの記述
- [ ] GitHub About の Website（`gh repo edit --homepage`）

置き換え不要（実害なし）:

- `src/face/*.js` / `src/tracking/recognizers.js` 内の `/tomari-guruguru/` は
  **コメントのみ**。実体は `import.meta.env.BASE_URL`（Vite が `base` から注入）で
  動くので機能には影響しない。気になればコメントも直す（任意）。

連動しないもの（混同注意）:

- `package.json` の `"name"` とローカルフォルダ名はリポジトリ名と独立。
- 本家所有の `docs/` 内の URL は触らない（[02](02-フォーク元の取り込み.md) 参照）。

置き換え対象の洗い出し:

```bash
grep -rniI "tomari-guruguru" \
  --include="*.js" --include="*.mjs" --include="*.json" \
  --include="*.md" --include="*.html" --include="*.bat" . \
  | grep -v node_modules | grep -v package-lock
```

## upstream マージとの関係（重要）

`vite.config.js` の `base: '/tomari-guruguru/'` は **いまは本家と同じ文字列**なので
衝突しない。**リネーム後は base が本家と食い違う**ため、本家が `vite.config.js` を
触るたびに衝突しやすくなる。

対策: base の上書きを `vite.fork.js`（`mergeConfig` 側）へ移し、`vite.config.js` の
`upstreamConfig` は本家と字面一致を保つ。こうすると本家の `vite.config.js` を
そのまま受けられて衝突が減る。

```js
// vite.fork.js 側で base を上書きする例（本家 vite.config.js は触らない）
// fork: Pages のリポジトリ名に追従させる base。
base: command === 'build' ? '/<新リポジトリ名>/' : '/',
```

リネームするなら、この base 移設も同時にやるのがおすすめ。

## リネーム手順

```bash
# 1. GitHub でリネーム（Web UI: Settings → Repository name、または gh）
gh repo rename <新リポジトリ名> --repo tommie-jp/tomari-guruguru

# 2. ローカル remote を更新（リダイレクトは効くが、明示が安全）
git remote set-url origin https://github.com/tommie-jp/<新リポジトリ名>.git

# 3. コード内の base 群を一括更新（上のチェックリスト）

# 4. 検証
npm run build         # camera.html / tracking.html が dist/ に出るか
npm run verify:pages  # Pages 向けの参照パス検証
npm run preview       # /<新リポジトリ名>/camera.html で開くか目視

# 5. Pages 再デプロイ後、GitHub About の Website を更新
gh repo edit tommie-jp/<新リポジトリ名> \
  --homepage "https://tommie-jp.github.io/<新リポジトリ名>/camera.html"
```

## 注意

- リネーム前に upstream マージや未マージ PR を片付けておくと、参照URLの混乱が減る。
- 旧URLのリダイレクトは、**旧名で新しいリポジトリを後から作ると無効化**される。
  旧名で別リポジトリを作らないこと。
