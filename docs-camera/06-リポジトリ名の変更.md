# リポジトリ名の変更（リネーム）記録

`tomari-guruguru` → **`guruguru-avatar`** へのリネーム記録（2026-06-17 実施）。
影響範囲・手順・upstream マージとの関係をまとめたメモ。

関連: [02-フォーク元の取り込み.md](02-フォーク元の取り込み.md) / [01-使い方.md](01-使い方.md)

## 結論

- フォークでもリネームは自由。**フォーク関係も `upstream` からの pull も壊れない**。
  GitHub が旧URL→新URLのリダイレクトを張る。
- ただしこのプロジェクトは **GitHub Pages のベースパス** がコード中に埋め込まれている。
  リネームで Pages URL が `tommie-jp.github.io/guruguru-avatar/` に変わるので、
  base 系の更新と再デプロイが必須。
- GitHub に「表示名」は無い。**リポジトリ名＝プロジェクト名**。

## 壊れるもの・壊れないもの

- 壊れない: フォーク関係、`upstream` remote、star / issue / PR、ローカル clone
  （いずれも GitHub のリダイレクトで追従）。
- 要更新: GitHub Pages の URL（`tommie-jp.github.io/guruguru-avatar/`）と、それに紐づく
  base path 群（下記チェックリスト）。

## upstream マージとの関係（重要）

本家 `vite.config.js` の `base: '/tomari-guruguru/'` は**本家と同じ文字列のまま**残す。
fork 側の base 上書きは `vite.fork.js`（`mergeConfig` の第2引数＝fork が勝つ）に集約した。
これで本家が `vite.config.js` を触っても 3-way マージがそのまま当たり、衝突しにくい。

```js
// vite.fork.js 側で base を上書き（本家 vite.config.js は触らない）。
// fork: Pages のリポジトリ名に追従させる base。
base: command === 'build' ? '/guruguru-avatar/' : '/',
```

## リネーム手順（実施した内容）

```bash
# 1. GitHub でリネーム
gh repo rename guruguru-avatar --repo tommie-jp/tomari-guruguru

# 2. ローカル remote を更新（リダイレクトは効くが、明示が安全）
git remote set-url origin https://github.com/tommie-jp/guruguru-avatar.git

# 3. コード内の base 群を一括更新（上のチェックリスト）

# 4. 検証
npm run build         # camera.html / tracking.html が dist/ に出るか
npm run verify:pages  # Pages 向けの参照パス検証
npm run preview       # /guruguru-avatar/camera.html で開くか目視

# 5. Pages 再デプロイ後、GitHub About の Website を更新
gh repo edit tommie-jp/guruguru-avatar \
  --homepage "https://tommie-jp.github.io/guruguru-avatar/camera.html"
```

## 注意

- リネーム前に upstream マージや未マージ PR を片付けておくと、参照URLの混乱が減る。
- 旧URLのリダイレクトは、**旧名で新しいリポジトリを後から作ると無効化**される。
  旧名で別リポジトリを作らないこと。
