# テスト（Vitest）

ユニットテストの実行方法と、これから増やすときの計画メモ。
フレームワークは **Vitest**（`package.json` の `devDependencies` / `"test"` スクリプト）で導入済み。
追加インストールなしで動く。

関連: [doTest.sh](../doTest.sh) / [07-バージョン管理.md](07-バージョン管理.md)

## 使い方（`./doTest.sh`）

`guruguru-avatar/` で実行する。中身は Vitest の薄いラッパー（`npm test` と同じ実体）。

| コマンド | 動作 |
| --- | --- |
| `./doTest.sh` | 全テストを1回実行（CI 向け・`vitest run`） |
| `./doTest.sh watch` | 変更を監視して自動再実行（開発中・`vitest`） |
| `./doTest.sh <pattern>` | ファイル名フィルタで一部だけ（例 `./doTest.sh obs`） |
| `./doTest.sh -h` | ヘルプ |

`npm test`（= `vitest run`）でも同じだが、`watch` と絞り込みを1コマンドにまとめてある。

## いま動いているテスト

純関数を中心にカバー（2026-06 時点で 17 ファイル / 158 件）。例:

- `src/obs-mode.test.js` — `?obs` の解析（三状態フラグ・rx 既定 ON）
- `src/relay-mode.test.js` — `?tx` / `?rx` / `?relay` の解析
- `src/face/*.test.js` — `avatar-state` / `apply-state` / `state-codec` / `relay-client` ほか

UI（React コンポーネント）やカメラ・描画は対象外。**URL 解析や状態変換のような副作用のない純関数**を優先的にテストしている。

## 仕組み

- テストファイルは実装の隣に置く（`src/**/*.test.js`）。
- 設定は `vite.config.js` を Vitest がそのまま読む（専用の `vitest.config.js` は無し）。
- 実行は Node 上（jsdom 等のブラウザ環境は今は使っていない＝純関数前提）。

## 段階的に増やす計画（今はやらない）

最初は「全実行・監視・絞り込み」だけで足りる。必要になったら次の順で足す。

1. **カバレッジ**

   ```bash
   npm i -D @vitest/coverage-v8
   ```

   その後 `./doTest.sh coverage` を `vitest run --coverage` として追加する
   （パッケージ未導入のまま `--coverage` を付けると対話プロンプトが出るので、導入とセットで実装する）。

2. **CI 連携**

   `pages.yml` のデプロイ前に `./doTest.sh` を挟み、グリーンでなければデプロイしない
   （[doDeploy.sh](../doDeploy.sh) の流れに組み込む）。

3. **E2E**

   描画・カメラ・OBS 受信（`?rx`）のような実ブラウザ挙動は Playwright で別レイヤにする。
   ユニットテストとは目的が違うので、必要が出てから最小構成で導入する。
   MCP Playwright は対話用で決定的なスイートに向かないため、E2E は **`@playwright/test` を
   別プロジェクトで**立てる。

## E2E でカメラをどう扱うか

WSL2 / CI には**カメラデバイスが無い**ので、`getUserMedia()` は `NotFoundError` で失敗する
（MCP Playwright で「いつもカメラが失敗する」のはこれ）。実カメラは使わず、次の2方針で回避する。

### 方針A: `?rx` でカメラを回避（描画系・推奨）

`?rx`（consumer）モードは [relay-mode.js](../src/relay-mode.js) のとおり**カメラを起動せず、
受信した状態フレームだけで描画**する。`state` フレームを WS（`npm run relay`）経由で注入すれば、
**カメラ無しで決定的に**描画系（sheet `A`〜`F` 切替・まばたき・口・transform）を検証できる。
推論層（`deriveFaceSignals` など）は既にユニットテスト済みなので、ここが E2E の高 ROI ゾーン。

### 方針B: フェイクカメラ（tx 経路を通したいときだけ）

Chromium のフェイクメディア用フラグで `getUserMedia` を通す。

| フラグ | 役割 |
| --- | --- |
| `--use-fake-ui-for-media-stream` | カメラ許可プロンプトを自動承認 |
| `--use-fake-device-for-media-stream` | 合成映像（転がる模様）を流す。`getUserMedia` は通る |
| `--use-file-for-fake-video-capture=face.y4m` | 指定した動画をカメラ映像として流す（ループ再生） |

注意: デフォルトの合成映像は**ただの模様**なので **MediaPipe は顔を検出できない**
（`target` が動かず tx テストにならない）。顔トラッキングまで見るなら、顔の動画を Y4M に変換して流す。

```bash
# 顔クリップを Y4M（無圧縮 YUV）へ。Chrome がこのファイルをループ再生する
ffmpeg -i face.mp4 -pix_fmt yuv420p -an e2e/fixtures/face.y4m
```

`@playwright/test` での設定例:

```js
// playwright.config.js
export default {
  use: {
    permissions: ['camera'],
    launchOptions: {
      args: [
        '--use-fake-ui-for-media-stream',
        '--use-fake-device-for-media-stream',
        // 顔トラッキングまで見るときだけ↓を足す
        '--use-file-for-fake-video-capture=' + require('path').resolve('e2e/fixtures/face.y4m'),
      ],
    },
  },
}
```

方針 B の MediaPipe 出力は機種・モデルでブレて非決定的なので、まずは方針 A（`?rx` 注入）を優先し、
B は tx 経路を実ブラウザで通す必要が出てから導入する。

## 推論をダミー MediaPipe でテストする（カメラ不要）

「カメラを使う推論そのもの」はテストしない。代わりに **MediaPipe をダミー化**して、推論より
下流のロジックをカメラ・WASM 無しで回す。このコードベースは MediaPipe を2つのきれいな seam で
隔離しているので差し込みは容易。差し込む層で2案ある。

> 注意（重複を避ける）: `deriveFaceSignals` / `avatar-state` / `apply-state` / `state-codec` は
> 純関数で**すでにユニットテスト済み**（`*.test.js` と `pipeline.test.js`）。ダミーで同じ所を
> 再検証しても価値は薄い。ダミーが本当に効くのは [use-face-pose.js](../src/face/use-face-pose.js)
> の**未カバーなグルー**——detector のライフサイクル（`status` の loading→running→error 遷移）と、
> `applySignals` の**顔ロスト時「向きは据え置き・他は中立へ」ルール**。

### 案A: 実 result を録画して再生（推論変換まで通す）

`deriveFaceSignals` の入力 `result` は MediaPipe の生データで、内訳は次の3つ:

| フィールド | 中身 | 使い道 |
| --- | --- | --- |
| `facialTransformationMatrixes[0].data` | 4x4 行列(16 float) | 顔向き。**これが無い＝顔ロスト** |
| `faceLandmarks[0]` | 478 点の `{x,y,z}` | 位置・見かけサイズ |
| `faceBlendshapes[0].categories` | `{categoryName, score}` の配列 | 口・目・表情 |

478 点を手書きするのは非現実的なので、**一度だけ実カメラで動かして `result` を JSON に保存し、
テストで再生する**（フィクスチャ化）。録画は dev でのみ、`detectForVideo` の直後に `result` を
`JSON.stringify` してログ/保存する一時タップを挟むのが手軽（[face-detector-main.js](../src/face/face-detector-main.js)
の `landmarker.detectForVideo(...)` の戻りが対象）。代表となる数フレームだけ残す。

保存ケース（成功・失敗の両方）:

- **成功・正面**（`result-front.json`）→ `faceDetected:true`・`x≈0`・`y≈0`
- **成功・横向き/口開け/まばたき** → それぞれ `x≠0` / `mouth` 高 / `eyesClosed` 高 を assert
- **失敗・顔なし**（`result-noface.json`）→ `facialTransformationMatrixes` が空。これは
  **録画不要**で `{}` や `{ "facialTransformationMatrixes": [] }` を置くだけで作れる
  → `faceDetected:false`＋中立リセットを検証できる

フィクスチャは `src/face/__fixtures__/result-*.json` に置く。使い方は2通り:

```js
// (1) 変換だけ確認: 録画 result を純関数にそのまま流す（derive-face-signals.test.js の延長）
import front from './__fixtures__/result-front.json';
expect(deriveFaceSignals(front).faceDetected).toBe(true);
```

```js
// (2) detector 経由で確認: FaceLandmarker をモックして録画 result を順に返す
vi.mock('./face-landmarker', () => ({
  createFaceLandmarker: async () => {
    let i = 0;
    const frames = [front, front, noface];      // 録画した result の並び
    return { detectForVideo: () => frames[Math.min(i++, frames.length - 1)], close() {} };
  },
}));
```

注意: フィクスチャは**モデル版（現在 `0.10.35`）に紐づく**。モデルを上げたら録り直す。
`result` は数 KB/フレームになるので代表数枚に絞る。

### 案B: signals を台本で流す（detector を DI・グルー狙い）

推論変換（`deriveFaceSignals`＝既済み）を飛ばし、detector を丸ごとフェイクして `signals` を
直接返す。録画も landmark も要らず、「顔あり → ロスト」のような**状態遷移を台本で**作れるので、
`useFacePose` のグルー検証に向く。`useFacePose` に detector を DI できるようにする（既定は本物）:

```js
// 既定は本物。テストのときだけ差し替える
const { createDetector = createFaceDetector, /* ... */ } = opts;
detector = await createDetector({ wasmPath, modelPath }, { preferWorker });
```

```js
// テスト用ダミー detector（signals を台本で返す）
export function createScriptedDetector(frames) {
  let i = 0;
  return {
    engine: 'fake',
    ready: Promise.resolve(),
    detect: () => frames[Math.min(i++, frames.length - 1)],
    close() {},
  };
}
```

### 隠れたコスト

案A(2) も案B も、`useFacePose`（React フック）を実際に回すには **jsdom 環境＋偽 `<video>`
（`readyState` / `currentTime` / `requestVideoFrameCallback`）＋ `startWebcam` のフェイク**が要る。
コストはダミーより**この jsdom ハーネス**の方。案A(1)（純関数に流すだけ）は jsdom 不要で最も安い。

### 段取り（安い順）

1. `applySignals` のロスト中立ルールを純関数に抽出してユニット（jsdom もダミーも不要・最優先）。
2. 描画系は方針A（`?rx` 注入）。
3. 変換まで実データで通したい → **案A(1)**（録画 result を `deriveFaceSignals` に流す）。
4. フックのライフサイクル/ロスト遷移まで通したい → 案A(2) か **案B**＋ jsdom ハーネス。
